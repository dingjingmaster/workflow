# AGENTS.md

> Agent 行为指南，旨在减少常见的 LLM 编码错误、代码重复、代码冗余、代码错误等。可根据需要与项目特定说明合并。

**权衡：** 这些指南偏向谨慎而非速度。对于琐碎任务，请自行判断。
**权衡：** 考虑到产品文档和需求文档可能逐步增大，根据实际对文档按结构进行拆分，最后分别在 `docs/overview-product.md` 和 `docs/overview-product-dev.md` 中创建关联链接。

## 0. 需求分级

**根据需求规模决定流程裁剪，避免小需求走重流程。**

| 级别 | 判断标准 | 流程 | 文档要求 |
|------|---------|------|---------|
| L1-微小 | 改文案、修小bug、调样式 | 直接编码 → 补 summary | 仅 summary |
| L2-小型 | 单模块小功能，不影响现有架构 | 简化 Research + Plan + Code + Summary | Research 和 Plan 都需要，但可简化；产品/开发文档仅更新变更部分 |
| L3-中型 | 跨模块功能，需要协调多个模块 | 完整流程 | 全部文档 |
| L4-大型 | 新系统、架构变更、多团队协作 | 完整流程 + 额外评审 | 全部文档 + 专项评审记录 |

**分级判断要点**：
- 拿到需求后先判断级别，有疑问时默认升级一级
- L1/L2 级别可跳过三角色审视中的部分检查项（见第7节）
- 任何级别如果编码中发现问题升级，需补齐对应级别文档

## 1. 明确需求

**需求确认、需求拆分、调研相似产品，生成产品设计文档**

> 本节定义"做什么"的原则。具体执行步骤见第8节行动指南。

接到用户需求后：
- 明确背景、目的、具体功能。如果不确定，就提问
- 拆分需求，如果你觉得需求不够详细，则将需求细分
- 找相似产品，分析其功能和实现方式，做技术可行性调研（考虑联网搜索）。有借鉴或需要调整的，提出来
- 确认需求后先修改产品设计文档（默认在 `docs/overview-product.md` 中，没有则根据模版 `.dj-agent/overview-product-template.md` 创建）
- 每次修改产品文档后要进行三角色审视（见第7节），有问题及时修改，有疑问提出来，直到修改满意为止

## 2. 开发文档

**分析需求，明确功能点、技术选型、模块划分、项目架构设计，形成开发文档**

> 本节定义"怎么做"的原则。具体执行步骤见第8节行动指南。

- 分析需求，明确基于现有产品功能的修改范围
- 技术选型，根据需求和调研结果，选择合适的的技术栈，确保较少依赖、较低成本、较高的可维护性、可扩展性等
- 将功能点划分为多个模块，每个模块负责不同的功能，确保模块之间的解耦和可维护性高，同时考虑模块之间的依赖关系和交互方式
- 确认技术选型后先修改开发文档（默认在 `docs/overview-product-dev.md` 中，没有则根据模版 `.dj-agent/overview-product-dev-template.md` 创建）
- 每次修改开发文档后要进行三角色审视（见第7节），有问题及时修改，有疑问提出来，直到修改满意为止

注意：**确保开发文档与产品设计文档保持一致，避免出现功能不一致或技术选型错误的情况**

## 3. 编码前先思考

**不要假设。不要隐藏困惑。揭示权衡。**

在实现之前：
- 明确陈述你的假设。如果不确定，就提问。
- 如果存在多种解释，请全部列出——不要默默选择。
- 如果有更简单的方法，说出来。在必要时提出反对意见。
- 如果有不清楚的地方，停下来。说明困惑之处。提问。

## 4. 简洁优先

**用最少的代码解决问题。不要做推测性的事情。**

- 不要添加超出要求的功能。
- 不要为仅使用一次的代码创建抽象。
- 不要添加未被请求的"灵活性"或"可配置性"。
- 不要为不可能发生的场景做错误处理。
- 如果你写了200行代码但50行就能解决，重写它。

问问自己："高级工程师会觉得这过于复杂吗？"如果是，简化它。

## 5. 精准修改

**只触碰必须修改的地方。只清理自己造成的混乱。**

编辑现有代码时：
- 不要"改进"相邻代码、注释或格式。
- 不要重构没有问题的东西。
- 匹配现有风格，即使你会用不同的方式。
- 如果发现无关的死代码，提及它——不要删除它。

当你的修改产生孤儿代码时：
- 移除**你的修改**导致不再使用的导入/变量/函数。
- 除非被要求，否则不要移除预先存在的死代码。

检验标准：每一行修改都应该能直接追溯到用户的请求。

## 6. 目标驱动执行

**定义成功标准。循环直到验证通过。**

将任务转化为可验证的目标：
- "添加验证" → "为无效输入编写测试，然后让它们通过"
- "修复bug" → "编写一个复现该bug的测试，然后让它通过"
- "重构X" → "确保重构前后测试都通过"

对于多步骤任务，简述计划：
```
1. [步骤] → 验证：[检查项]
2. [步骤] → 验证：[检查项]
3. [步骤] → 验证：[检查项]
```

强大的成功标准让你能独立循环。薄弱的标准（"让它能工作"）需要不断澄清。

---

**这些指南生效的标志是：** diff中不必要的变更更少、因过度复杂导致的重写更少、澄清问题出现在实现之前而非犯错之后。

## 7. 三角色审视

**每个阶段完成后，从三个角色审视结论。不同需求级别可裁剪检查项。**

### 7.1 检查清单

| 角色 | 检查项 | 适用级别 |
|------|--------|---------|
| **高级产品** | ① 是否真正解决了用户问题？ | L2+ |
| | ② 是否存在过度设计或功能蔓延？ | L2+ |
| | ③ 与现有功能是否冲突或重复？ | L2+ |
| | ④ 边界场景和异常流程是否覆盖？ | L3+ |
| | ⑤ 用户体验是否一致？ | L3+ |
| **高级架构师** | ① 方案是否可扩展？是否为未来留有余地？ | L2+ |
| | ② 是否引入了不必要的依赖或复杂度？ | L2+ |
| | ③ 模块边界是否清晰？职责是否单一？ | L2+ |
| | ④ 是否有单点故障或性能瓶颈风险？ | L3+ |
| | ⑤ 与现有架构的兼容性如何？ | L3+ |
| **高级工程师** | ① 实现是否简洁？有无更简单的方案？ | L1+ |
| | ② 是否有性能隐患（N+1、内存泄漏等）？ | L1+ |
| | ③ 错误处理是否充分且不过度？ | L2+ |
| | ④ 测试是否可验证成功标准？ | L2+ |
| | ⑤ 代码是否符合项目现有风格和规范？ | L1+ |

### 7.2 审视方式

- **L1**：仅做高级工程师检查项 ①②⑤
- **L2**：高级产品 ①②③ + 高级架构师 ①②③ + 高级工程师全部
- **L3/L4**：全部检查项
- 审视发现问题 → 立即修正 → 重新审视该角色 → 通过后继续

## 8. 行动指南

**标准流程：Research → Plan → Code → Summary → 更新总文档**

> 第1-2节定义了"做什么"和"怎么做"的原则，本节定义具体的执行步骤和规范。

### 8.1 文档规范

同一需求的所有文档使用统一ID，存放在 `docs/dev/` 目录。需求多了以后，在 `docs/dev/README.md` 维护索引：

```
docs/dev/
├── README.md                       # 索引：000001=用户登录, 000002=支付...
├── 000001-research-用户登录.md
├── 000001-plan-用户登录.md
├── 000001-summary-用户登录.md
└── ...
```

**文档模板章节标注**：模板中标注 🔴必填 / 🟡按需 / ⚪选填，避免为填模板而填模板。原则：与当前需求无关的章节直接跳过，标注"不适用"即可。

### 8.2 执行步骤

| 阶段 | 对应原则 | 动作 | 模板 | 完成标志 | 总文档 |
|------|---------|------|------|---------|--------|
| **Research** | 第1节 | 调研相似方案，给出推荐 | `001-research-template.md` | 推荐方案明确 + 三角色审视 | 不更新 |
| **Plan** | 第2节 | 分解任务，制定计划 | `002-plan-template.md` | 任务清单清晰 + 三角色审视 | 不更新 |
| **Code** | 第3-6节 | 按计划编码，写测试 | - | 测试通过 | 不更新 |
| **Summary** | - | 记录决策、经验、债务 | `003-summary-template.md` | 总结完整 + 三角色审视 | **此时更新** |

**按需求级别裁剪**：

| 级别 | Research | Plan | Code | Summary |
|------|----------|------|------|---------|
| L1 | 跳过 | 跳过 | 直接编码 | 补写（简化版） |
| L2 | 简化（可内嵌Plan） | 必需 | 按计划 | 必需 |
| L3 | 必需 | 必需 | 按计划 | 必需 |
| L4 | 必需 + 专项调研 | 必需 + 评审 | 按计划 | 必需 + 评审 |

### 8.3 需求变更处理

| 变更程度 | 处理方式 |
|---------|---------|
| 小调整（不影响方案） | 直接更新 Plan 文档，标注变更原因 |
| 大变更（推翻方案） | 回到 Research，在原ID下追加调研或创建新ID |

### 8.4 流程回退机制

**编码过程中发现问题时，按以下规则回退：**

| 发现的问题 | 回退到 | 处理方式 |
|-----------|--------|---------|
| Plan 中的技术方案不可行 | Plan | 更新技术方案，标注原因，重新三角色审视 |
| Plan 中遗漏了关键模块/依赖 | Plan | 补充模块划分，评估影响范围，重新三角色审视 |
| Research 遗漏了关键信息 | Research | 补充调研，评估是否影响 Plan，如影响则同步更新 Plan |
| 编码中发现需求理解偏差 | Research | 回到需求确认，与用户澄清，评估影响后决定是否需要新 ID |

**回退原则**：
- 回退时在文档中标注回退原因和影响范围，不删除原有记录
- 小范围回退（不影响整体方案）可直接在当前文档中补充，无需新建 ID
- 大范围回退（推翻方案）按 8.3 大变更处理

### 8.5 上下文管理

**LLM 上下文窗口有限，需按需加载文档，避免信息过载。**

**加载策略**：

| 场景 | 加载内容 | 说明 |
|------|---------|------|
| 新需求启动 | `docs/dev/README.md` 索引 + 相关 summary | summary 是历史需求的压缩上下文 |
| Research 阶段 | 当前需求的 research 文档 + 产品文档相关章节 | 只加载与需求相关的章节 |
| Plan 阶段 | 当前需求的 research + plan 文档 + 开发文档相关章节 | 按需加载，不加载无关模块 |
| Code 阶段 | 当前 plan 文档 + 相关源码 | 优先加载 plan 中的任务清单和技术要点 |
| Summary 阶段 | 当前需求的全部文档 | 编码完成后的完整回顾 |

**文档瘦身原则**：
- 产品文档和开发文档按模块拆分章节，每个需求只更新相关章节
- Summary 文档是"压缩上下文"，新需求启动时优先加载相关 summary 而非原始 research/plan
- 当单个文档超过 500 行时，考虑按模块拆分为独立文件

### 8.6 快速检查清单

- [ ] 判断需求级别（L1-L4）
- [ ] 分配ID（检查 `docs/dev/` 最大序号+1，L1 可后补）
- [ ] 调研文档 → `docs/dev/[ID]-research-[简述].md` → 三角色审视（L2+）
- [ ] 计划文档 → `docs/dev/[ID]-plan-[简述].md` → 三角色审视（L2+）
- [ ] 编码完成，测试通过
- [ ] 总结文档 → `docs/dev/[ID]-summary-[简述].md` → 三角色审视
- [ ] 更新 `docs/overview-product.md` 和 `docs/overview-product-dev.md`，添加链接（L2+）
- [ ] 更新 `docs/dev/README.md` 索引
