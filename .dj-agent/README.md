# AGENTS.md

> Agent 行为指南，旨在减少常见的 LLM 编码错误、代码重复、代码冗余、代码错误等。可根据需要与项目特定说明合并。

**权衡：** 这些指南偏向谨慎而非速度。对于琐碎任务，请自行判断。
**权衡：** 考虑到产品文档和需求文档可能逐步增大，根据实际对文档按结构进行拆分，最后分别在 `docs/overview-product.md` 和 `docs/overview-product-dev.md` 中创建关联链接。

## 0. 明确需求

**需求确认、需求拆分、调研相似产品，生成产品设计文档**

接到用户需求后：
- 明确背景、目的、具体功能。如果不确定，就提问
- 拆分需求，如果你觉得需求不够详细，则将需求细分
- 找相似产品，分析其功能和实现方式，做技术可行性调研（考虑联网搜索）。有借鉴或需要调整的，提出来
- 确认需求后先修改产品设计文档（默认在 `docs/overview-product.md` 中，没有则根据模版 `.dj-agent/overview-product-template.md` 创建）
- 每次修改产品文档后要进行三个角度思考，从高级产品、高级架构师、高级工程师角度进行思考需求合合理性、技术可行性、对整体功能的影响，有问题及时修改，有疑问提出来，直到修改满意为止

## 1. 开发文档

**分析需求，明确功能点、技术选型、模块划分、项目架构设计，形成开发文档**

- 分析需求，明确基于现有产品功能的修改范围
- 技术选型，根据需求和调研结果，选择合适的的技术栈，确保较少依赖、较低成本、较高的可维护性、可扩展性等
- 将功能点划分为多个模块，每个模块负责不同的功能，确保模块之间的解耦和可维护性高，同时考虑模块之间的依赖关系和交互方式
- 确认技术选型后先修改开发文档（默认在 `docs/overview-product-dev.md` 中，没有则根据模版 `.dj-agent/overview-product-dev-template.md` 创建）
- 每次修改开发文档后要进行三个角度思考，从高级产品、高级架构师、高级工程师角度进行思考需求合合理性、技术可行性、模块划分是否合理，有问题及时修改，有疑问提出来，直到修改满意为止。

注意：**确保开发文档与产品设计文档保持一致，避免出现功能不一致或技术选型错误的情况**

## 2. 编码前先思考

**不要假设。不要隐藏困惑。揭示权衡。**

在实现之前：
- 明确陈述你的假设。如果不确定，就提问。
- 如果存在多种解释，请全部列出——不要默默选择。
- 如果有更简单的方法，说出来。在必要时提出反对意见。
- 如果有不清楚的地方，停下来。说明困惑之处。提问。

## 3. 简洁优先

**用最少的代码解决问题。不要做推测性的事情。**

- 不要添加超出要求的功能。
- 不要为仅使用一次的代码创建抽象。
- 不要添加未被请求的"灵活性"或"可配置性"。
- 不要为不可能发生的场景做错误处理。
- 如果你写了200行代码但50行就能解决，重写它。

问问自己："高级工程师会觉得这过于复杂吗？"如果是，简化它。

## 4. 精准修改

**只触碰必须修改的地方。只清理自己造成的混乱。**

编辑现有代码时：
- 不要"改进"相邻代码、注释或格式。
- 不要重构没有问题的东西。
- 匹配现有风格，即使你会用不同的方式。
- 如果发现无关的死代码，提及它——不要删除它。

当你的修改产生孤儿代码时：
- 移除**你的修改**导致不再使用的导入/变量/函数。
- 除非被要求，否则不要移除预先存在的死代码。

检验标准：每一行修改都应该能直接追溯到用户的请求。

## 5. 目标驱动执行

**定义成功标准。循环直到验证通过。**

将任务转化为可验证的目标：
- "添加验证" → "为无效输入编写测试，然后让它们通过"
- "修复bug" → "编写一个复现该bug的测试，然后让它通过"
- "重构X" → "确保重构前后测试都通过"

对于多步骤任务，简述计划：
```
1. [步骤] → 验证：[检查项]
2. [步骤] → 验证：[检查项]
3. [步骤] → 验证：[检查项]
```

强大的成功标准让你能独立循环。薄弱的标准（"让它能工作"）需要不断澄清。

---

**这些指南生效的标志是：** diff中不必要的变更更少、因过度复杂导致的重写更少、澄清问题出现在实现之前而非犯错之后。

## 6. 行动指南

**标准流程：Research → Plan → Code → Summary → 更新总文档**

### 6.1 文档规范

同一需求的所有文档使用统一ID，存放在 `docs/dev/` 目录。需求多了以后，在 `docs/dev/README.md` 维护索引：

```
docs/dev/
├── README.md                       # 索引：000001=用户登录, 000002=支付...
├── 000001-research-用户登录.md
├── 000001-plan-用户登录.md
├── 000001-summary-用户登录.md
└── ...
```

### 6.2 执行步骤

| 阶段 | 动作 | 模板 | 完成标志 | 总文档 |
|------|------|------|---------|--------|
| **Research** | 调研相似方案，给出推荐 | `001-research-template.md` | 推荐方案明确 + 三角色审视 | 不更新 |
| **Plan** | 分解任务，制定计划 | `002-plan-template.md` | 任务清单清晰 + 三角色审视 | 不更新 |
| **Code** | 按计划编码，写测试 | - | 测试通过 | 不更新 |
| **Summary** | 记录决策、经验、债务 | `003-summary-template.md` | 总结完整 + 三角色审视 | **此时更新** |

> **三角色审视**：每个阶段完成后，从高级产品、高级架构师、高级工程师三个角度审视结论，有问题及时修正。

### 6.3 需求变更处理

| 变更程度 | 处理方式 |
|---------|---------|
| 小调整（不影响方案） | 直接更新 Plan 文档，标注变更原因 |
| 大变更（推翻方案） | 回到 Research，在原ID下追加调研或创建新ID |

### 6.4 快速检查清单

- [ ] 分配ID（检查 `docs/dev/` 最大序号+1）
- [ ] 调研文档 → `docs/dev/[ID]-research-[简述].md` → 三角色审视
- [ ] 计划文档 → `docs/dev/[ID]-plan-[简述].md` → 三角色审视
- [ ] 编码完成，测试通过
- [ ] 总结文档 → `docs/dev/[ID]-summary-[简述].md` → 三角色审视
- [ ] 更新 `docs/overview-product.md` 和 `docs/overview-product-dev.md`，添加链接
- [ ] 更新 `docs/dev/README.md` 索引
