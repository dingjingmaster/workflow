# 05-safety-gates.md

> 导航：入口 `AGENTS.md` ｜ 上一节 `00-routing.md` ｜ 下一节 `10-research.md`
>
> 安全门禁：本文件始终加载，优先级高于按级别裁剪规则。若与其它分片冲突，执行更保守的规则。

## 1. 规则语义

- “必须”：没有满足前不得继续。
- “禁止”：除非用户明确授权，否则不得执行。
- “需要用户确认”：先说明原因、影响范围、风险和替代方案，获得明确同意后再执行。
- 不确定是否属于高风险时，按高风险处理。

## 2. 默认安全姿态

- 默认先只读分析，再决定是否修改。
- 修改文件前必须明确：目标、范围、预期影响、验证方式。
- 每一行变更必须能追溯到当前需求、bug 根因或验证要求。
- 不允许顺手重构、批量格式化、清理历史遗留代码或调整无关文档。
- 不允许用大范围重写替代局部修复。
- 发现需求不清、影响范围扩大、验证不可落地或风险不可评估时，必须暂停并询问用户。

## 3. 命令权限分级

| 级别 | 说明 | 示例 | 规则 |
|------|------|------|------|
| C0-只读 | 只读取仓库、进程或环境信息，不修改状态 | `pwd`、`ls`、`rg`、`sed -n`、`git status`、`git diff`、`git show` | 可直接执行 |
| C1-工作区内写入 | 只修改当前需求范围内文件或运行定向验证 | `apply_patch`、定向 formatter、定向测试、项目本地 build/test | 执行前必须明确目标文件、范围和验证目的 |
| C2-需确认 | 可能改变权限、依赖、暂存区、提交历史以外的仓库状态，或需要网络/外部资源 | `chmod`、`mv`、`cp -f`、依赖安装、网络下载、`docker compose down`、`git add`、`git commit`、`git push` | 必须先获得用户明确确认 |
| C3-高危 | 可能删除、覆盖、破坏数据，改变系统/设备/生产状态，或难以自动回滚 | 见“破坏性操作禁令” | 未获用户明确授权时禁止执行 |

补充规则：
- 不确定命令属于哪个级别时，按更高风险级别处理。
- 不执行未知脚本或未知二进制的写入型操作；必须先只读检查脚本内容和调用参数。
- 不使用 `git add -A` 或 `git add .`，只允许暂存本次需求明确相关文件。
- 用户要求提交或推送时，也必须先说明将暂存/提交/推送的文件范围。

## 4. 破坏性操作禁令

未获得用户明确授权，禁止执行下列操作：

- 删除、覆盖或迁移用户数据。
- `rm -rf`、大范围 `rm`、递归覆盖、批量重命名。
- `git reset --hard`、`git checkout --`、`git clean`、`git rebase`、`git push --force`。
- `sudo`、系统级包安装/卸载、修改系统级配置。
- `systemctl stop/restart`、`kill -9`、停止关键进程或服务。
- `dd`、`mkfs`、分区、格式化、刷机、写固件、修改启动项。
- `mount`、`umount`、修改设备节点、写入裸设备。
- `insmod`、`modprobe`、`rmmod`、加载/卸载内核模块。
- 生产环境部署、线上数据库写操作、不可自动回滚的数据迁移。
- 对未知脚本或二进制执行写入型操作。

## 5. 工作区保护

- 编码前必须检查工作区状态。
- 不得覆盖、回退或删除用户已有修改。
- 如果目标文件已有未确认修改，必须先阅读并基于现状小心合并。
- 修改只限当前需求必要文件；新增文件必须说明用途。
- 格式化只限本次修改涉及的文件或项目明确要求的最小范围。
- 生成文件、锁文件、依赖文件只有在本次需求直接需要时才可修改。

工作区处理协议：
- 修改前必须执行 `git status --short`。
- 目标文件已修改时，必须查看该文件 diff，区分“用户已有修改”和“本次计划修改”。
- 修改后必须能说明每个变更文件属于：本次修改 / 用户已有修改 / 生成文件。
- 不得通过格式化、重命名或大范围移动掩盖逻辑修改。
- 发现用户已有修改与任务冲突时，暂停并说明冲突，不得自行覆盖、回退或重写。

## 6. 高风险开发门禁

满足任一条件，最低按 L2 执行；跨模块、影响运行时稳定性或涉及系统副作用时按 L3+ 执行：

- C/C++ 逻辑变更。
- Rust `unsafe` 或 FFI 变更。
- 内核、驱动、系统调用、设备 I/O、文件系统、网络协议相关变更。
- 客户端与内核/驱动/系统服务的边界变更。
- 并发、锁、线程、协程、异步调度、原子操作相关变更。
- 内存生命周期、指针、缓冲区、序列化/反序列化相关变更。
- ABI/API、协议、共享数据结构、配置格式变更。
- 构建系统、链接参数、编译选项、交叉编译、发布包内容变更。
- 持久化数据、迁移、升级/回滚、安全、权限、性能关键路径变更。

L0/L1 不适用于上述场景，除非只是注释、文档、拼写或无行为变化的小修。

## 7. Bug 证据等级

| 等级 | 判定标准 | 允许动作 |
|------|----------|----------|
| E0-无证据 | 无稳定复现、无日志/堆栈、无明确触发条件 | 只能只读调查、补充日志或询问用户，不得改核心逻辑 |
| E1-弱证据 | 有现象描述、日志、堆栈或用户反馈，但不能稳定复现 | 可做定位、加防御性验证或低风险观测改动；核心修复需说明假设和残余风险 |
| E2-可复现 | 有稳定复现步骤、失败命令或明确触发输入 | 可进入修复；必须修复后回归验证 |
| E3-自动化复现 | 有失败单测、集成测试、脚本或最小复现工程 | 优先先保留失败用例，再实施最小修复并证明通过 |

证据等级规则：
- Bug 修复开始时必须标注证据等级。
- E0 不得进入 Code 阶段修改核心逻辑。
- E1 修改必须保持最小范围，并在 Summary 记录未覆盖风险。
- E2/E3 必须记录修复前后的复现/回归结果。

## 8. 证据优先

- 需求实现必须先定义可验证成功标准。
- Bug 修复必须先记录现象、复现方式或足够的定位证据。
- 不能复现且证据不足时，不得猜测性修改核心逻辑。
- 验证失败时必须先判断是否由本次修改导致，再决定修复、回退到 Plan 或询问用户。

## 9. 停止条件

出现以下情况必须暂停：

- 需要执行破坏性操作但用户未授权。
- 需要执行 C2/C3 命令但用户未确认。
- 继续修改可能覆盖用户已有工作。
- Bug 证据等级为 E0，但任务要求修改核心逻辑。
- 发现需求理解与代码现实冲突。
- 实现需要超出当前分级的架构、接口、数据或部署变更。
- 无法给出可复现验证方式，且风险不是 L0。
