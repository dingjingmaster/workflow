# 20-plan.md

> 导航：上一节 `10-research.md` ｜ 下一节 `30-code.md`
>
> 本阶段定义“怎么做”：功能点拆解、技术选型、模块边界、任务清单与验证项。

## 1. 输入与目标

- 输入：Research 结论、相关产品/开发文档、现有代码结构。
- 目标：形成可直接执行的任务计划与验证标准，并判断是否需要更新开发文档。

## 2. 执行动作

- 分析变更范围，明确基于现有功能要修改的模块。
- 明确最大修改范围、禁止触碰范围和需要用户确认的操作。
- 按风险矩阵确认最终级别；按命令权限分级列出 C2/C3 操作。
- 做技术选型：优先少依赖、低成本、可维护、可扩展。
- 划分模块职责与依赖关系，确保边界清晰、职责单一。
- 将任务拆成可验证步骤，每步都附带验证项；多步任务记录状态。
- 对命中的安全门禁逐项给出应对和验证方式。
- 判断是否影响架构、公共接口、数据模型、依赖、部署、长期通用验证方式、项目级验证入口或发布验证标准。
- 对 L2+ 或已创建轻量 Summary 的任务，执行计划是进度单一事实源：开始步骤前标记为“进行中”，验证通过后标记为“完成”，阻塞时标记为“阻塞”并写明原因。

## 3. 文档规则

- L2：需求实现写入 `docs/dev/[ID]-task-[简述].md` 的“执行计划”；bug 修复写入 `docs/dev/[ID]-fix-[简述].md` 的“修复执行计划”。验证方式写入计划表，验证结果在 Summary/验证记录阶段补充。
- L3/L4：创建独立 `docs/dev/[ID]-plan-[简述].md`。
- 只有影响架构、公共接口、数据模型、依赖、部署、长期通用验证方式、项目级验证入口或发布验证标准时，才需要更新 `docs/overview-product-dev.md`；事实稳定且确需立即沉淀时可在本阶段更新，否则在 Summary 阶段更新。
- 如果开发文档不存在，先按 `.dj-agent/overview-product-dev-template.md` 创建骨架，再只补相关章节。

## 4. 执行清单边界

- 不创建独立 todo list 模板；Plan 或 L2 文档中的“执行计划/修复执行计划”就是当前需求的执行清单。
- L0/L1 不维护持久 todo；只在回复或轻量 Summary 中说明结果与验证。
- L2+ 的执行计划必须包含：步骤、修改内容、验证方式、状态。
- 状态只用于当前需求推进，建议取值：待开始 / 进行中 / 完成 / 阻塞。
- 按计划完成任一修改步骤后，必须在继续下一阶段或提交前更新对应状态和验证结果摘要；不得只在最终回复中口头说明。
- 执行计划不承载长期 backlog、跨需求排期或发散想法；此类内容只在用户明确要求时另建专题文档。
- 计划变更时，更新执行计划并在 Summary 记录计划偏差，不另起一份 todo。

## 5. 完成标志

- 任务清单清晰，执行顺序合理，验证项可落地。
- 风险门禁、风险矩阵、命令权限、修改边界和禁止触碰范围已明确。
- 已明确是否需要更新开发文档；事实稳定且确需立即沉淀时已更新，否则在 Summary 阶段更新。
- 当前级别和阶段要求多角色审视时，审视已通过。

## 6. 计划模板建议

| 步骤 | 修改内容 | 验证方式 | 状态 |
|------|----------|----------|------|
| 1 | [描述] | [命令/步骤] | 待开始 |
| 2 | [描述] | [命令/步骤] | 待开始 |

## 7. 约束

- 计划只包含当前需求所需内容，不扩展未来需求。
- 发现技术不可行或遗漏关键模块时，按附录回退机制处理。
