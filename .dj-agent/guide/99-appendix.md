## 1. 需求变更处理

| 变更程度 | 处理方式 |
|---------|---------|
| 小调整（不影响方案） | 直接更新 Plan 文档，标注变更原因 |
| 大变更（推翻方案） | 回到 Research，在原 ID 下追加调研或创建新 ID |

## 2. 流程回退机制

编码过程中发现问题时，按以下规则回退：

| 发现的问题 | 回退到 | 处理方式 |
|-----------|--------|---------|
| Plan 中技术方案不可行 | Plan | 更新技术方案，标注原因，重新三角色审视 |
| Plan 中遗漏关键模块/依赖 | Plan | 补充模块划分，评估影响范围，重新三角色审视 |
| Research 遗漏关键信息 | Research | 补充调研，评估是否影响 Plan；如影响则同步更新 Plan |
| 编码中发现需求理解偏差 | Research | 回到需求确认，与用户澄清，评估是否需要新 ID |

回退原则：
- 回退时标注原因与影响范围，不删除原有记录。
- 小范围回退（不影响整体方案）可在当前文档补充，无需新建 ID。
- 大范围回退（推翻方案）按“需求变更处理”执行。

## 3. 文档规范补充

- 同一需求的文档统一使用同一 ID，存放在 `docs/dev/`。
- 索引由 `docs/dev/README.md` 维护。
- 模板章节标注（必填/按需/选填）遵循“与需求无关则标注不适用”。

## 4. 回退记录模板（建议）

```text

- 需求 ID：
- 触发阶段：Research / Plan / Code / Summary
- 回退原因：
- 影响范围（模块/文档/测试）：
- 处理动作：
- 是否升级级别：否 / 是（Lx -> Ly）
- 结论与后续：
```
