> 回退机制、需求变更处理与文档规范补充。

## 1. 需求变更处理

| 变更程度 | 处理方式 |
|---------|---------|
| 小调整（不影响方案） | 直接更新 Plan 或轻量任务文档，标注变更原因 |
| 大变更（推翻方案） | 回到 Research，在原 ID 下追加调研或创建新 ID |

## 2. 流程回退机制

编码过程中发现问题时，按以下规则回退：

| 发现的问题 | 回退到 | 处理方式 |
|-----------|--------|---------|
| Plan 中技术方案不可行 | Plan | 更新技术方案，标注原因，重新多角色审视 |
| Plan 中遗漏关键模块/依赖 | Plan | 补充模块划分，评估影响范围，重新多角色审视 |
| Research 遗漏关键信息 | Research | 补充调研，评估是否影响 Plan；如影响则同步更新 Plan |
| 编码中发现需求理解偏差 | Research | 回到需求确认，与用户澄清，评估是否需要新 ID |
| 任务影响范围扩大 | Routing | 重新分级，补齐对应级别文档与审视项 |
| 需要破坏性操作或系统级副作用 | Plan / 用户确认 | 暂停执行，说明原因、影响、替代方案，获得明确授权后继续 |
| Bug 无法复现且证据不足 | Research | 补充复现、日志、调用链或最小证据；不足时不得猜测性改核心逻辑 |
| Bug 证据等级为 E0 | Research / 用户确认 | 只能补充证据、增加观测或询问用户，不得修改核心逻辑 |
| 目标文件存在用户未确认修改 | Plan / 用户确认 | 暂停并说明冲突，基于现状调整方案，不得覆盖或回退 |
| 计划中出现 C2/C3 命令 | Plan / 用户确认 | 说明命令、目的、影响和替代方案，获得明确确认后继续 |

回退原则：
- 回退时标注原因与影响范围，不删除原有记录。
- 小范围回退（不影响整体方案）可在当前文档补充，无需新建 ID。
- 大范围回退（推翻方案）按“需求变更处理”执行。

## 3. 文档规范补充

- 每次任务、问题修复或需求变更只要新增文档，必须分配文件编号。
- 文件编号使用从 `1` 开始递增的正整数，全局只在 `docs/dev/` 下递增，不按类型分别编号，不要求固定位数。
- 分配编号前必须同时检查 `docs/dev/README.md` 索引和 `docs/dev/` 现有文件名，取最大编号 + 1。
- 编号一旦分配不得复用；任务取消、方案废弃或回退重开，也在索引中保留记录并标注状态。
- 同一需求的多份文档统一使用同一编号，存放在 `docs/dev/`。
- 文件命名格式：`docs/dev/[N]-[type]-[slug].md`。
- `type` 取值：
  - `task`：需求实现、普通任务、L2 单文件记录。
  - `fix`：问题修复、缺陷定位与修复记录。
  - `research`：L3/L4 调研文档。
  - `plan`：L3/L4 计划文档。
  - `summary`：L1+ 总结或 L3/L4 结论文档。
- `slug` 使用短横线命名，优先英文小写；无法准确英文表达时使用简短拼音或中文短语，但必须稳定可读。
- L1 如创建轻量 Summary，使用 `docs/dev/[N]-summary-[slug].md`。
- L2 默认使用 `docs/dev/[N]-task-[slug].md`；若是 bug/problem 修复，优先使用 `docs/dev/[N]-fix-[slug].md` 和 `.dj-agent/fix-lite-template.md`。
- L3/L4 使用同一编号创建 Research、Plan、Summary 三类文档。
- 索引由 `docs/dev/README.md` 维护。
- 新增、取消、完成、拆分、合并文档时，必须同步更新 `docs/dev/README.md`。
- 模板章节标注（必填/按需/选填）遵循“与需求无关则标注不适用”。

```text
- 是否升级级别：否 / 是（Lx -> Ly）
- 结论与后续：
```
