# 00-routing.md

> 导航：上一节 `AGENTS.md` ｜ 下一节 `10-research.md`
>
> 需求分级、阶段裁剪、触发条件与默认加载入口。安全门禁见 `05-safety-gates.md`，始终优先于本文件。

## 1. 需求分级

| 级别 | 判断标准 | 流程 | 文档要求 |
|------|---------|------|---------|
| L0-快速 | 拼写、注释、单行配置、无行为变化的小修，且未命中安全门禁高风险项 | 直接修改 -> 最小验证 -> 回复说明 | 不建文档 |
| L1-微小 | 改文案、修小 bug、调样式、局部低风险行为修正，且未命中安全门禁高风险项 | 直接编码 -> 验证 -> 简化 Summary | 默认不建独立文档；必要时更新索引或模块文档 |
| L2-小型 | 单模块小功能，不影响现有架构 | 简化 Research + Plan + Code + Summary | 默认更新模块文档和索引；复杂任务才用 `task-lite`/`fix-lite` 独立记录 |
| L3-中型 | 跨模块功能，需要协调多个模块 | 完整流程 | 同一编号的 Research / Plan / Summary 文档 |
| L4-大型 | 新系统、架构变更、多团队协作 | 完整流程 + 按需专项评审 | 同一编号的 Research / Plan / Summary 文档；存在专项评审、审计或外部评审时追加 Review 文档 |

分级判断要点：
- 拿到需求后先判断级别；影响范围不清先快速探查，仍不确定或涉及取舍/风险时再提问。
- 不因实现细节不确定自动升级；只有影响范围、接口、数据、安全、性能或架构变化时才升级。
- 命中 `05-safety-gates.md` 高风险开发门禁时，最低按 L2；跨模块、影响运行时稳定性或涉及系统副作用时按 L3+。
- 分级按“风险矩阵”取最高级别，不得只按改动文件数降级。
- Bug 修复如果无法复现且缺少足够定位证据，不得进入 Code 阶段猜测性修改核心逻辑。
- L1/L2 级别可跳过多角色审视中的部分检查项（见 `50-review-checklist.md`）。
- 任一级别如果编码中发现问题升级，需补齐对应级别的索引、模块文档或独立文档要求。

风险矩阵：

| 维度 | L0/L1 | L2 | L3 | L4 |
|------|-------|----|----|----|
| 修改范围 | 单点、无行为或低风险行为变化 | 单模块 | 跨模块/跨进程/跨语言边界 | 新系统或长期架构变化 |
| 运行时风险 | 不影响稳定性和数据 | 局部失败可控 | 影响关键链路、并发、内存、性能或权限 | 可能影响系统/设备/生产稳定性 |
| 接口/数据 | 无接口、协议、配置、数据变化 | 私有接口或局部配置 | 公共 API/ABI/协议/共享数据结构 | 迁移、兼容策略、发布/回滚策略变化 |
| 可回滚性 | 可直接撤销 | 可通过代码回退恢复 | 回退需协调多个模块或版本 | 回退困难或存在不可逆副作用 |
| 可观测性 | 代码阅读或单命令验证 | 本地自动化可验证 | 需要集成、压力、兼容或环境验证 | 需要部署、监控、灰度或专项验证 |

## 2. 阶段与加载策略

标准流程：Research -> Plan -> Code -> Summary -> 按需更新总文档

加载策略：
- 新需求启动：`AGENTS.md` + `AGENTS.project.md`（如存在）+ 本文件 + `05-safety-gates.md` + 已存在的 `docs/dev/README.md` 本地索引；索引不存在时先跳过加载，首次需要记录 L1+ 变更前用 `.dj-agent/dev-index-template.md` 初始化。
- 相关上下文加载：先只读 `docs/dev/README.md`，按需求关键词、模块、文件路径、接口/API、错误现象或历史编号匹配；优先展开匹配的 `docs/dev/modules/[module].md`，再按需最多展开 3-5 个最相关的独立 Summary、task 或 fix 文档；不确定相关性时只保留索引信息，不展开全文。
- Research 阶段：追加相关模块文档、当前独立需求文档的 Research 相关章节或独立 Research 文档 + 产品文档相关章节。
- Plan 阶段：追加相关模块文档、当前独立需求文档的 Research/Plan 相关章节或独立 Research/Plan 文档 + 开发文档相关章节。
- Code 阶段：追加当前对话计划、相关模块文档或独立 Plan 文档 + 相关源码 + 验证约定。
- Summary 阶段：加载本次修改涉及的模块文档、索引和当前独立需求文档（如有），回顾并沉淀结论。

项目覆盖层策略：
- `AGENTS.project.md` 是项目级持久覆盖层，必须与 `AGENTS.md` 同级；存在时必须在每个新需求启动时加载，L0/L1 也不能跳过。
- `AGENTS.project.md` 默认只读加载；不得自动创建、更新、整理、覆盖或删除该文件。
- 只有用户明确指明“加入/更新/删除 `AGENTS.project.md` 内容”时，才允许按 `.dj-agent/AGENTS.project-template.md` 创建或修改该文件。
- 用户在工作中给出本项目长期有效的配置、安全边界、禁止触碰范围、固定验证命令或提交/部署约束，但未明确要求写入 `AGENTS.project.md` 时，仅在当前对话中遵守并提示可加入该文件，不得自动落盘。
- 仅当前任务有效的临时要求不写入 `AGENTS.project.md`；如果无法判断是否长期有效，可询问用户是否需要加入该文件。
- 共享工作流更新、模板同步或批量修改不得覆盖 `AGENTS.project.md`；若确需调整其中规则，必须能追溯到用户明确要求。

文档瘦身原则：
- `docs/dev/README.md` 是任务与模块变更索引，记录每次需要沉淀的 L1+ 变更；`docs/dev/modules/` 是默认长期上下文，按模块维护当前行为、接口、验证方式、故障模式和修复记录。
- L1/L2 默认不新建独立 `docs/dev/[N]-*.md`；功能开发和 bug 修复优先更新对应模块文档，并在 `docs/dev/README.md` 追加索引行。
- 只有满足以下任一条件时，才新增独立编号文档：L3/L4；复杂 bug 需要完整证据链；跨模块方案取舍需要复盘；专项评审/审计；用户明确要求；模块文档会因过程细节膨胀且信息不适合长期保留。
- 新增独立任务、问题、调研、计划、总结或评审文档时，必须按 `docs/dev/[N]-[type]-[slug].md` 编号命名，并更新 `docs/dev/README.md`。
- 确需更新产品文档或开发文档时，按模块拆分章节，每个需求只更新相关章节。
- 模块文档和独立 Summary/task/fix 文档都可作为压缩上下文；新需求优先加载模块文档，再按索引加载相关独立文档。
- 当单个模块文档超过 500 行时，按子模块拆分为多个 `docs/dev/modules/[module]-[topic].md`。

L1 轻量 Summary 边界：
- 纯文案、注释、格式、拼写或无行为变化的小改，不建文档，最终回复说明结果与验证。
- 改代码且影响用户可见行为、配置、验证方式或后续维护判断时，优先更新模块文档和索引；只有没有合适模块文档且确需独立复盘时，才创建本地轻量 Summary。
- L1 bug 修复有 E1-E3 证据时，优先在模块文档的修复记录或故障模式中沉淀；E0 仍按安全门禁只读调查或补证据。

## 3. 阶段触发条件（强制）

满足任一条件，最低按 L2 执行：
- 命中 `05-safety-gates.md` 的高风险开发门禁。
- 修复需要改动 C/C++、Rust `unsafe`、内核/驱动、并发、内存生命周期、ABI/API/协议或构建链接逻辑。
- 需要新增持久化验证脚本、跨模块自动化测试或专门复现工程来证明行为；仅为小范围修复补充轻量回归测试时不单独触发升级。

满足任一条件，自动升级到 L3（必要时升级到 L4）：
- 跨 2 个及以上模块协同改动。
- 修改公共接口/协议（API、事件、共享数据结构）。
- 引入新依赖或基础设施变化（缓存、消息队列、数据库迁移等）。
- 影响关键链路的性能、稳定性或安全性。
- 需要迁移历史数据或改变部署/运维方式。
- 涉及系统副作用、内核模块、设备 I/O、发布包、升级/回滚或生产环境行为。

同时满足全部条件，可按 L2 简化执行：
- 单模块内改动。
- 无公共接口变化。
- 无新依赖与基础设施变化。
- 文件数量不作为单独升级依据；改动文件较多时，已按风险矩阵复核且未出现跨模块、接口、数据或架构风险。
- 验证方式明确且可在本地完成。
- 不涉及破坏性操作或需要用户确认的系统级操作。

同时满足全部条件，可按 L1/L0 执行：
- 不改变公共接口、数据结构和长期产品行为。
- 改动范围小，能够通过代码阅读或单个验证命令确认。
- 不需要用户做产品或技术取舍。
- 未命中 `05-safety-gates.md` 的高风险开发门禁。
- Bug 修复已有明确复现或明确根因证据。

审视清单加载时机：
- L0：不加载 `50-review-checklist.md`。
- L1：仅 Summary 阶段加载 `50-review-checklist.md` 并执行 L1 裁剪项。
- L2：Plan、Summary 阶段加载 `50-review-checklist.md` 并执行 L2 裁剪项。
- L3/L4：Research、Plan、Code、Summary 每阶段加载 `50-review-checklist.md` 并执行全量项。

Review 文档只记录专项评审、审计或外部评审结论；各阶段多角色审视按上述加载时机执行，未发生专项评审时不创建 Review 文档。

## 4. 按级别裁剪

| 级别 | Research | Plan | Code | Summary |
|------|----------|------|------|---------|
| L0 | 跳过 | 跳过 | 直接修改 | 回复中说明 |
| L1 | 跳过 | 跳过 | 直接编码 | 轻量记录或回复中说明 |
| L2 | 简化并记录到对话、索引或模块文档 | 必需但可轻量 | 按计划 | 写入模块文档和索引；复杂任务写入独立 task/fix |
| L3 | 必需 | 必需 | 按计划 | 必需 |
| L4 | 必需，按需专项调研 | 必需，按需专项评审 | 按计划 | 必需，按需记录专项评审结论 |

## 5. 快速检查清单

- [ ] 判断需求级别（L0-L4）
- [ ] 加载 `AGENTS.project.md`（如存在）；除非用户明确指明加入、更新或删除该文件内容，否则不得修改
- [ ] 检查是否命中 `05-safety-gates.md`
- [ ] 按风险矩阵确认最终级别
- [ ] 检查是否涉及 C2/C3 命令或 E0/E1 bug 证据
- [ ] L1+ 如需记录本地上下文，优先更新 `docs/dev/README.md` 和 `docs/dev/modules/[module].md`；索引不存在时先初始化
- [ ] 如需新增独立本地文档，分配从 `1` 开始递增的编号（索引存在时检查 `docs/dev/README.md`；同时检查 `docs/dev/` 文件名最大序号 + 1；索引不存在时先初始化）
- [ ] L2：默认更新模块文档和索引；只有复杂任务/复杂 bug 才创建 `docs/dev/[N]-task-[slug].md` 或 `docs/dev/[N]-fix-[slug].md`
- [ ] L3/L4：使用同一 `N` 创建 Research、Plan、Summary 文档；L4 按需创建 Review 文档
- [ ] 编码完成，验证通过
- [ ] 更新当前模块文档或独立任务文档进度：执行计划状态、验证结果、Summary 或轻量总结
- [ ] 按需更新 `docs/overview-product.md` 与 `docs/overview-product-dev.md`
- [ ] L1+ 如更新本地上下文，更新 `docs/dev/README.md` 索引，并在最终回复说明
- [ ] 如用户明确要求提交，或 `AGENTS.project.md` 明确声明本项目默认自动提交，则按 `.dj-agent/commit-message-template.md` 提交本次相关变更；否则最终回复说明未提交
