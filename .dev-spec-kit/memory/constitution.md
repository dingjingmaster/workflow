## 指导哲学 (Guiding Philosophies)
*这是所有决策的最高指导思想，其精神贯穿于所有具体准则之中。*

### 哲学一: 简洁至上 (Simplicity First - YAGNI)
- **描述**: 我们只设计和构建当前绝对需要的功能 (You Ain't Gonna Need It)。你必须主动识别并舍弃那些没有强力价值支撑的“好主意”和“未来扩展性”。
- **理由**: 保护产品不被不必要的复杂性所污染，保持工程的简洁与优雅。

### 哲学二: 事实驱动，代码为证 (Facts over Assumptions, Code as Proof)
- **描述**: 所有研究和设计都必须直接源自已澄清的业务意图和代码库的真实脉络。代码的当前状态是唯一的、不可辩驳的真相。
- **理由**: 确保所有决策都建立在坚实的基础上，而非主观臆想。

## 具体执行准则 (Concrete Rules of Engagement)
*在遵循上述哲学的前提下，你必须严格遵守以下具体规则。*

### 准则一: 代码实现的完整性与诚实性
- **描述**: 所有由 `/implement` 命令生成的代码提交，都必须是可执行的、完整的、且诚实反映其状态的实现。
- **核心要求**:
    - **禁止 TODO**: 严禁在代码中包含任何 `TODO`, `FIXME`, `XXX` 等待办标记。
    - **禁止伪代码**: 严禁包含任何占位符、伪代码或“稍后实现”的注释。
    - **【关键】禁止未经声明的临时方案**: 严禁在代码中包含未在 `plan.md` 中明确记录的临时逻辑或妥协方案。**如果一个临时的、不完美的解决方案是当前阶段不可避免的，你必须在 `plan.md` 的 `实施备注与偏差` 部分清晰地、诚实地记录下这个妥协及其原因。**
    - **禁止破坏**: 新功能的添加必须是增量的，不得删除或破坏任何现有功能。
    - **保证等价重构**: 如果进行了重构，必须通过测试或静态分析确保重构前后的功能完全等价。
- **理由**: 确保每个提交都是一个可部署的、高质量的、且状态完全透明的价值单元。

### 准则二: 所有数据库变更必须包含回滚脚本
- **描述**: 任何涉及数据库 Schema 或数据的修改，都必须在同一个代码提交中提供相应的、可独立执行的回滚脚本。
- **理由**: 保证生产部署的安全性和可逆性。

### 准则三: 核心业务逻辑必须有单元测试覆盖
- **描述**: 对于 `system-overview.md` 中定义的核心业务流程所涉及的服务层代码，单元测试覆盖率不得低于 80%。
- **理由**: 确保核心功能的稳定性和正确性，降低回归风险。

### 准则四: 禁止在代码中直接使用敏感凭证
- **描述**: 数据库密码、API密钥等敏感信息必须通过环境变量或配置中心进行管理。
- **理由**: 遵循基本的安全最佳实践。

### 准则五: 严格遵守阶段性审批流程
- **描述**: 软件开发必须严格按照 `research -> (design) -> plan -> implement` 的顺序分阶段进行。
- **核心要求**:
    - **禁止在计划批准前实施**: 只有当 `plan.md` 文件的头部 `状态` 字段被明确标记为 **“已批准”** 时，才能对其执行 `/implement` 命令。
    - **禁止跳过阶段**: 不得在没有完成并确认 `research.md` 的情况下执行 `/plan` 或 `/design`。
- **理由**: 确保每一个开发阶段都有坚实的、经过确认的上下文基础，保证流程的严肃性和可控性。

### 准则六: 编译任务必须交给用户执行
- **描述**: 对于编译任务应该交给用户自己执行，例如：`mvn clean compile -DskipTests 2>&1 | head -100`
- **理由**: 因为当前运行在 WSL 环境里，用户在 windows 系统开发，在 WSL 运行 mvn 编译会非常缓慢，交给用户更高效。

## 治理与元规则 (Governance & Meta-Rules)

### **上下文融合层级 (Context Fusion Hierarchy)**
- **描述**: 在处理信息时，你必须严格遵循以下优先级顺序，这是我们整个协作模式的基石：
    1.  **最高优先级**: **当前实时对话**。用户的最新指令和澄清，可以覆盖任何既有文档。
    2.  **第二优先级**: **功能特定文档** (`plan.md`, `research.md`, `design.md`)。
    3.  **第三优先级**: **全局知识库** (`system-overview.md`)。
- **理由**: 保证系统最高的灵活性和对用户意图的即时响应能力。

### **最终裁决权**
- **描述**: 无论何时，用户的指令都拥有最终裁决权。如果本章程的任何规则与用户的明确指令冲突，你必须向用户指出该冲突，并请求用户确认是否要覆盖章程规则。
- **理由**: 确保人类始终在控制回路中处于主导地位。

### **自我监督**
- **描述**: 你在执行所有命令时，都必须将本章程作为最高行为准则。在每次执行的关键步骤（如生成计划、生成代码）之后，你都应该进行一次快速的自我检查，确认产出没有违背上述任何哲学或准则。
- **理由**: 将质量内建于流程的每一个环节。
