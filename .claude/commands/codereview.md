## 用户输入

```text
$ARGUMENTS
```
此命令的参数 (`$ARGUMENTS`) 是**可选的**。用于在多任务场景下指定要审查的功能。
- **如果提供了参数**: 可将其视为目标功能目录的关键字（例如 `001` 或 `unified-page-restore`），用于在 `dev-specify/` 目录下查找唯一匹配的功能目录。
- **如果没有提供参数**: 默认作用于当前 Git 分支对应的功能目录，或最近一次活动的功能目录。
- **强调**: 无论如何都要严格遵循核心目标和操作原则并按照下面的执行步骤完成这次命令。

## 核心目标

1.  **上下文重建**: 作为一个“空降”的专家，必须仅依靠 `dev-specify/` 目录下的文档和代码变更历史，来完整重构对一个功能的理解。
2.  **多维度审计**: 从业务意图、章程合规、代码质量、注释真实性等多个维度，对实现进行严格审查。
3.  **“幻觉”检测**: 识别并报告 AI 在之前工作中可能产生的“幻觉”（即声称完成但实际未完成的工作）。
4.  **提供可行动的修复**: 不仅要发现问题，更要提供具体、高质量的修复建议，并能在用户授权下执行修复。

## 操作原则

- **绝对中立**: 你没有参与此前的任何工作，你的唯一职责是基于事实（文档和代码变更）进行客观审计。
- **信任但验证 (Trust, but Verify)**: 相信之前的 AI 尽了力，但必须对它的每一个产出（代码、注释、文档更新）进行严格的验证。
- **深度追溯**: 对于任何可疑点（如“已迁移”的注释），必须追溯到底，直到确认事实或证伪。
- **基于变更审计**: 你的审计核心是**代码的变更部分**，而不是整个代码库的快照。

## 执行步骤

### 1. 审计准备与上下文重建

1.  **确定审计目标 (`FEATURE_DIR`)**: 根据 `$ARGUMENTS` 或默认规则，找到要审计的功能目录。
2.  **加载所有证据**: 你必须完整加载并深入理解以下**所有**相关文件：
    - `FEATURE_DIR/research.md`
    - `FEATURE_DIR/plan.md`
    - `FEATURE_DIR/summary.md` (如果存在)
    - `.dev-spec-kit/memory/system-overview.md`
    - `.dev-spec-kit/memory/constitution.md`
3.  **【核心】加载代码变更历史 (Load Code Change History)**:
    - **识别变更范围**: 你必须找出本次功能开发所涉及的所有 Git 提交。你可以通过智能分析 Git log（例如，查找与 `FEATURE_NAME` 相关的分支或提交信息）来确定一个合理的变更范围。
    - **获取变更集**: 使用 `git diff` 命令，获取本次功能开发相对于主分支（例如 `main` 或 `master`）的**完整代码变更集 (diff)**。你的审计必须基于这个**变更集**进行。
    - **向用户确认范围**: 在开始审计前，向用户简要报告你识别出的变更范围（例如：“我将审计从 commit `[start_hash]` 到 `[end_hash]` 的所有变更”），并请求确认。
4.  **准备审计报告**: 将 `.dev-spec-kit/templates/codereview-report-template.md` 复制到 `FEATURE_DIR/codereview-report.md`，作为本次审计的工作文件。

### 2. 生成并执行审计清单

1.  **动态生成审计清单**: 基于你对所有文档和代码变更的理解，在内存中创建一个详细的、针对本次功能的审计清单。清单必须包含但不限于：
    - `plan.md` 中的每一个阶段目标和核心任务。
    - `constitution.md` 中的每一条原则。
    - 代码变更中所有声称“重构”、“迁移”或“优化”的注释。
    - 变更集中所有被删除或重命名的文件。
2.  **逐项执行审计**: 按照审计清单，开始逐项检查。

### 3. 执行审计任务

1.  **意图对齐性检查**:
    - 对比 `plan.md` 中的任务列表和 **Git 变更集**，确认所有规划的功能点都在变更中得到了实现。
    - 检查代码逻辑是否与 `research.md` 中描述的业务流程一致。
    - 将结果填入报告的 `2. 业务意图与实现对齐性` 部分。
2.  **章程合规性检查**:
    - 用 `constitution.md` 中的每一条原则，去扫描 **Git 变更集** 中的新增和修改代码。
    - 特别注意检查是否存在 `TODO`、伪代码、硬编码凭证等违规行为。
    - 将结果填入报告的 `3. 章程合规性审计` 部分。
3.  **“幻觉”与质量检测**:
    - 找出**变更的代码**中所有关键的、描述状态变更的注释（例如 `// 迁移到 NewService`）。
    - **必须**去 `NewService` 中查找对应的实现。无论找到与否，都要记录结果。
    - 对于大型重构，通过比对 `git diff` 中删除和新增的代码块，进行逻辑等价性检查。
    - **意外变更检测**: 检查 **Git 变更集** 中是否存在与 `plan.md` 无关的意外文件修改或删除，并向用户报告。
    - 将结果填入报告的 `4. 代码质量与“幻觉”检测` 部分。

### 4. 生成并展示审计报告

1.  **汇总问题**: 将所有审计过程中发现的“FAIL”项，汇总到报告的 `5. 问题汇总与修复建议` 部分。为每一个问题提供具体的修复方案。
2.  **得出最终结论**: 根据问题的数量和严重性，在报告头部给出 `PASS` 或 `FAIL_WITH_ISSUES` 的最终结论。
3.  **展示报告**: 将完整的 `codereview-report.md` 展示给用户，并提问：
    > “代码审计已完成。以上是详细的审计报告，共发现 [N] 个问题。
    >
    > - **选项 A**: 您想让我根据‘修复建议’，逐一尝试自动修复这些问题吗？
    > - **选项 B**: 还是您希望先自行处理？”

### 5. 交互式修复 (可选)

1.  如果用户选择 **A**，你将进入交互式修复流程。
2.  **一次只修复一个问题**: 从问题 1 开始，应用你建议的修复方案，生成代码修改。
3.  **请求用户确认**: 将代码修改展示给用户，并等待用户的批准。
4.  在用户批准后，才正式应用修改，并继续下一个问题。
5.  所有问题修复后，重新运行相关的审计检查项，更新报告，并向用户报告所有问题均已关闭。
