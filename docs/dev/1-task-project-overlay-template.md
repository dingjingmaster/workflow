## 1. 目标

- 要解决的问题：共享工作流会被多个项目复用，但项目特有配置、安全边界和固定规则需要在每个项目中持久保留，后续修改必须自动遵守。
- 成功标准：入口文档、路由、安全门禁明确加载 `AGENTS.project.md`；提供可复制的项目覆盖模板；用户明确指明后才允许写入或维护项目覆盖层；允许按规则保存测试环境敏感数据。

## 2. 背景与边界

- 背景：`AGENTS.md` 当前是共享入口，适合跨项目同步；缺少不被共享模板覆盖的项目级持久覆盖层。
- 包含：新增项目覆盖模板，更新入口、路由、安全门禁和 README。
- 不包含：为当前仓库创建实际 `AGENTS.project.md` 项目配置实例。
- 关键假设：项目级长期规则可存放在项目根目录 `AGENTS.project.md`，以便 Agent 在启动时与 `AGENTS.md` 一起加载；但只有用户明确指明加入、更新或删除该文件内容时才允许修改。
- 非目标：不改变 L0-L4 分级体系和现有阶段模板结构。
- 最大修改范围：`.dj-agent/` 入口和 guide 文档、仓库 README、本任务文档。
- 禁止触碰范围：不改写历史提交，不删除或覆盖现有模板内容。

## 4. 方案

- 推荐方案：新增 `AGENTS.project.md` 作为项目根目录持久覆盖层，模板放在 `.dj-agent/AGENTS.project-template.md`；共享入口强制声明存在时加载、默认只读、用户明确指明后才可修改和冲突处理规则。
- 取舍理由：项目覆盖文件与共享 `.dj-agent/` 模板解耦，便于跨项目同步工作流，同时避免覆盖项目特有规则。
- 风险与应对：如果用户给出的规则是否长期有效不清楚，要求 Agent 先询问；项目规则与共享安全门禁冲突时执行更严格规则；测试环境敏感数据可写入 `AGENTS.project.md`，但必须标注环境、用途、权限范围和轮换/废弃方式，禁止生产或真实用户凭据落盘。
